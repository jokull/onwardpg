# Phase classification

Every generated operation is placed on one side of exactly one application
deployment. This inventory records why former middle-phase work belongs in
expand, contract, an editable two-sided handoff, or a separate plan.

| Operation family | Classification | Required invariant | Evidence |
| --- | --- | --- | --- |
| New schemas, enums, tables, nullable columns, extensions, routines, triggers, ordinary indexes | Expand | Existing code does not lose an interface; hazards still describe locks, scans, and index builds | `TestCreatePlanConvergesOnPostgreSQL`, `TestRoutineAndTriggerLifecycleConvergesOnPostgreSQL` |
| New constraint on a new table | Expand | The table is itself not yet used by old code | `TestCreatePlanConvergesOnPostgreSQL` |
| New CHECK or foreign key on an existing table | Contract `ADD ... NOT VALID`, then separate validation | Old writers have drained; new writes are fenced before historical rows are scanned | `TestBuildNewCheckOnExistingTableStagesValidationInContract`, `TestNotValidForeignKeyCreateConvergesOnPostgreSQL` |
| New unique index over a newly added column on an existing table | Preparation decision, then contract | Production rows are explicitly declared clean or operator-authored cleanup and verification run after old writers drain | `TestBuildCapturesPreparationForUniqueIndexOnNewColumn` |
| Proven CHECK widening | Atomic replacement and validation in expand | Every old accepted row satisfies the desired predicate; both old and new values are accepted | `TestBuildCheckWideningIsExpandOnly`, `TestCheckWideningAcceptsLegacyAndNewWritesAfterExpand` |
| CHECK narrowing | Keep old CHECK in expand; stage, validate, and swap desired CHECK in contract | Legacy writes retain their old accepted domain until drain | `TestBuildCheckNarrowingKeepsOldEnforcementUntilContract` |
| Unknown or cross-name CHECK evolution | Drop old CHECK in expand; add `NOT VALID` and validate desired CHECK in contract | The overlap schema is deliberately looser; PostgreSQL does not reject either writer based on an unproven predicate relationship | `TestBuildUnknownCheckChangeUsesLooseExpandEnvelope`, `TestCrossNameCheckTransitionAcceptsLegacyAndNewWritesAfterExpand` |
| CHECK, FK, primary/unique constraint, exclusion, or standalone unique-index removal | Expand after enforcement-specific confirmation | Removing enforcement cannot reject legacy SQL and may be required for new writes; the exact lost guarantee is reported | `TestBuildRendersIfExistsForConstraintDrop`, `TestPartitionedInheritedCheckDropConvergesOnPostgreSQL` |
| Foreign-key mutation | Drop old FK in expand; add `NOT VALID` and validate desired FK in contract | Neither reference domain is allowed to reject the overlap writer; referential-action behavior change is explicit | `TestBuildForeignKeyMutationUsesLooseAcceptanceEnvelope` |
| Existing `NOT VALID` constraint validation | Expand | PostgreSQL already enforces the constraint for new rows; validation scans historical rows only | `TestNotValidConstraintValidationConvergesOnPostgreSQL`, `TestBuildValidatesNotValidConstraintWithoutRebuild` |
| New required column without a default | Expand nullable column; contract backfill pocket and `SET NOT NULL` | Old writers can omit the column before deployment; new code writes it; old writers drain before enforcement | `TestBuildStagesNewRequiredColumnWithoutDefault` |
| Nullable column becoming `NOT NULL` | Contract | Old writers that may produce nulls have drained; backfill assertions are true | `TestBuildStagesApplicationBackfillBeforeNotNullContract` |
| Drop `NOT NULL` | Expand | Relaxing the invariant does not break old code | `TestMutationPlanConvergesOnPostgreSQL` |
| Same-type column rename | Expand shadow column and trigger; explicit manual, single-transaction, or split-plan backfill strategy; contract assertion and native rename | Both names remain writable; divergent dual writes fail; equality is verified and pre-deployment writers drain before bridge removal | `TestBuildRequiresFingerprintBoundColumnRenameAnswer`, `TestColumnRenameOverlapBridgeConvergesOnPostgreSQL`, `TestColumnRenameWithExistingTriggerIsRejectedOnPostgreSQL` |
| Type or semantic conversion | Editable expand and contract pockets, or explicit split-plan blocker | Product code proves one deployed version works with both representations; onwardpg never invents conversion or precedence | `TestBuildRequiresAndRendersColumnMutationChoices`, `TestSemanticManualSQLHandoffRequiresEditedCloneVerificationOnPostgreSQL` |
| Supported table rename | Expand new-name updatable view; contract atomic view removal and physical rename | Tested view shape supports required old/new reads and writes; old callers drain before removal | `TestTableRenameConvergesOnPostgreSQL` |
| View, routine, trigger, policy, RLS, and privilege behavior changes | Contract after confirmation where required; explicit compatibility blocker for unbridged application identities | Pre-deployment callers no longer require old behavior and the one new release works before and after the change | `TestRoutineReplacementRequiresMaterializedViewRefreshContract`, `TestRoutineAndTriggerLifecycleConvergesOnPostgreSQL`, `TestRowSecurityPoliciesAndTablePrivilegesConvergeOnPostgreSQL` |
| Partition attach/detach or persistence change | Contract | Pre-deployment writers have drained; scans and access-exclusive locks are reviewed | `TestBuildRendersPartitionAttachDetachAndRejectsReconfiguration`, `TestPartitionAttachAndDetachConvergeOnPostgreSQL` |
| Product-specific partition reconfiguration | Contract edit pocket | Agent supplies exact data movement and verifies bounds; planner does not invent detach/attach ordering | `TestBuildRequiresFingerprintBoundManualContractForPartitionReconfiguration` |
| Materialized-view rebuild or refresh | Contract | Stale stored data and blocking refresh/rebuild behavior are explicitly accepted | `TestMaterializedViewCreateRebuildAndDropConvergeOnPostgreSQL`, `TestRoutineReplacementRequiresMaterializedViewRefreshContract` |
| Ordinary non-unique index replacement | Expand same-name replacement build; contract old-index cleanup | Replacement is usable before old index is removed; concurrent statements remain outside transactions | `TestBuildContinuouslyReplacesSameNameIndex`, `TestContinuousMaterializedViewAndLocalPartitionIndexReplacementConvergesOnPostgreSQL` |
| Proven weaker standalone unique index | Drop obsolete uniqueness and build desired index in expand | The structural key/predicate proof shows the desired enforcement accepts every old state | `TestBuildStandaloneUniqueRelaxationConvergesInExpand` |
| Unknown standalone unique-index replacement | Drop obsolete uniqueness in expand; restore desired index in contract | Old enforcement cannot reject new writes; the temporary duplicate window is explicit | `TestBuildUnknownUniqueIndexReplacementUsesLooseEnvelope` |
| Proven weaker UNIQUE or primary key | Swap to the key with additional parts in expand | The old key implies the desired weaker key; retained external dependents block | `TestBuildUniqueKeyRelaxationIsExpandOnly`, `TestBuildPrimaryKeyRelaxationIsExpandOnly` |
| Incomparable primary/unique-constraint key | Drop obsolete uniqueness in expand; restore desired constraint in contract | Neither key implies the other, so retaining the old key could reject a valid new write; retained external dependents block | `TestBuildIncomparableUniqueConstraintUsesLooseEnvelope` |
| Exclusion-constraint mutation | Drop obsolete exclusion enforcement in expand; restore desired constraint in contract | Without a bounded operator proof, neither constraint may reject an overlap writer; retained external dependents block | `TestBuildExclusionMutationUsesLooseAcceptanceEnvelope`, `TestExclusionConstraintCreateAndDropConvergeOnPostgreSQL` |
| Constraint-kind change under one name | Drop obsolete enforcement in expand; restore the desired kind in contract | Different enforcement semantics are incomparable unless separately proven; retained external dependents block | `TestBuildConstraintKindChangeUsesLooseAcceptanceEnvelope` |
| Constraint-backed index replacement | Expand replacement index; contract brief constraint swap | New index is ready before the enforced identity changes | `TestContinuousPrimaryAndUniqueConstraintReplacementConvergesOnPostgreSQL` |
| Add or change a same-type column default | Expand | A new writer may omit a required column only after its desired default exists; old INSERT syntax remains accepted | `TestBuildColumnDefaultPolaritySupportsBothWriters` |
| Remove a column default | Contract | Legacy writers that still omit the column have drained | `TestBuildColumnDefaultPolaritySupportsBothWriters` |
| Sequence `OWNED BY`, identity, serial, generated-expression, and sequence-parameter changes | Contract unless creating a wholly new object | Old code has drained before generation or cascade behavior changes; destructive identity removal is confirmed | `TestSequenceOwnedByTransitionsConvergeOnPostgreSQL`, `TestIdentityAddAllOptionsAndConfirmedDropConvergeOnPostgreSQL` |
| Approved data/interface cleanup (tables, columns, non-unique indexes, views, routines) | Contract | Destructive intent is fingerprint-bound and old code no longer uses the object | `TestApprovedDestructiveDropsConvergeOnPostgreSQL` |

Manual work is not a phase. A manual statement belongs in expand or contract
according to its timing. Transactional mode is orthogonal: either file may
contain multiple transactional and non-transactional batches.

Where the catalog cannot prove the invariant in this table, onwardpg must ask,
emit an editable pocket, or return unsupported. It must not relabel a direct
cutover as contract and call it a one-deployment strategy.
